Part 4 Documentation

All the code and files used for this app are contained inside the part4 folder. To connect to this app, either press the link in the sidebar, or you can directly connect to it by entering the URL http://localhost:8000/part4/ . I chose to name this app simply part4 rather then something like computer store to avoid confusion between the part3 computer store

Much of the funcitonality is the same as part3 from a user perspective, however much has changed behind the scenes. There is now an authentication system with the app. If a user tries to access this app or a page in this app without signing in, the user will automatically be redirected to the login page. At the login page, the user can either enter their credentials and log into the app, or they can choose to proceed to the create account screen, where the user will be automatically redirected to the welcome page of the app and signed in after creating an account. There is also now a logout button at the top navigation of the app which signs the user out and redirects to the login page.

What has changed behind the scenes is that we are no longer using cookies and we are required to use a database to store and update all our data. In the part4 app folder, you will notice the json data has been deleted and can no longer be used in the app. Instead, we will utilize django models to create our tables in our db and store our data. All the relevant Django models used in this assignment can be viewed in the models.py file in the part4 folder. The image above shows all the models used for the app. The Users model is already created by default as thanks to django, so all the user related data is stored in this default model. There is also a computers model, which is very similar to what was done for computers in the part3 json. There is a model for each part_name (cpu, ram, etc...), and a model for cart.

Because we are now needing to interact more with the backend, the command folder now contains many more commands, allowing the client to send more requests to the server. In the parts page of part4 we are now sending a get request which tells the server to run a query against the db to fetch all the relevant parts data, and send it back to the client in json format. In the computers section of part4, the exact same thing is done but now for computers. When a computer is added to cart, essentially now what happens is we send a request (command) to add an entry to the cart model, as shown above. This entry in the cart model contains all the info related to that computer the user has added, as well as the userID to allow the server to know which cart entry is for which user. So when the user presses the cart button in the navigation in part4, a query is run on the backend to fetch all cart entries that have the same userID as the user, and send that data to the client side to render to the browser. Upon deleting or purchasing computers, the cart entries are deleted from the model.
